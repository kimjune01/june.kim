### The problem before merge

[Blind, blind, merge](/blind-blind-merge) works when the input is sharp. Two models implement independently, complementary failures cancel, merge produces something better than either. But the method amplifies whatever you give it. Sharp spec → sharp code. Vague spec → two different kinds of wrong.

The original experiment hand-wrote the spec over two days. Twenty revisions. Fifteen design decisions. Where does a sharp spec come from?

### The volley

<img src="/assets/img/volley-workflow.svg" alt="Volley workflow: goal and repo feed you, you and reviewer bounce back and forth, fixed point when neither improves, output is a sharp testable claim that feeds blind-blind-merge" style="width:100%;"/>

State a goal. Two models bounce it between them until neither can improve it.

Claude drafts. Codex challenges. "What does 'improve ranking' mean?" "What metric?" "What's the baseline?" "What assumptions are you making?" Claude adjusts. Codex challenges again. The volley ends when the ball stops moving. You stated the goal. The models did the sharpening.

The output is whatever survives the exchange. For a spec, that's a document precise enough that implementation is mechanical. For a taxonomy, it's a grid where every cell has been checked. For a blog post, it's a draft where every claim has been challenged. The method doesn't care what's being volleyed. It cares that both sides are trying to land a better shot.

### What the volley does

You state the goal. Claude drafts a version. The draft has hidden assumptions, unstated dependencies, and vague success criteria. Codex surfaces them. Each round replaces a vague claim with a testable one. The volley converges when every claim is either testable or explicitly marked as an assumption.

What survives is the version that can't be improved by either model. What gets filtered is what sounded good but couldn't survive a return.

### Fixed point in two rounds

The convergence has the same shape as the "a bit" dampener in the [writing skills](/double-loop). Run humanize once: ten findings. Again: two. A third time: nothing to cut. The qualifier "a bit" dampens the skill to idempotency.

The volley works the same way. First round: codex challenges five assumptions. Second round: two clarifications. Third round: stable. Neither model finds anything to push back on. Two to three iterations to convergence, same as the skill dampener.

This is why it works as a one-liner. You don't specify the number of rounds. The rounds terminate when neither model can improve the shot.

### Example: gemini-cli state loss

A [gemini-cli](https://github.com/google-gemini/gemini-cli) bug: `setHistory()` was silently dropping conversation state during context compression. The goal: "fix state loss without regressions."

Volley round 1: codex identified three separate mutation paths that could corrupt state — `setHistory()` in `geminiChat.ts`, abort signal threading in `chatCompressionService.ts`, and a race in `clusterSummarizer.ts`. I would have fixed only the first.

Volley round 2: I pushed back that the abort signal was unrelated. Codex showed the call chain: compression triggers `setHistory()`, which triggers a re-render, which fires during an active `resolveDirty()` call. The abort signal isn't unrelated — it's the mechanism that prevents the race.

Round 3: stable. The [resulting fix](https://github.com/google-gemini/gemini-cli/commit/e6b7a1146) touched 8 files, added 138 lines of tests, and the PR passed gemini's own review. Three rounds, four hours wall-clock, zero revisions after merge. On a codebase I didn't write, with a bug that had three interacting causes I wouldn't have found alone.

### The workflow

1. **Goal**: you state it. *(checkpoint: does this goal make sense?)*
2. **Volley**: Claude and codex bounce until stable. *(checkpoint: does this spec match my intent?)*
3. **[Blind, blind, merge](/blind-blind-merge)**: two models implement independently.
4. **Merge**: complementary failures cancel. *(checkpoint: does the PR description match the spec?)*
5. **Tests**: verify against the volleyed criteria.

You attend to the checkpoints. Everything between them is automated. You never read the code. You read the spec, the prereg, and the PR description. Four gates, each a yes/no. The volley produces the spec. Blind-blind-merge produces the code. The tests verify the contract. Your job is to make sure each gate's output matches your intent.
